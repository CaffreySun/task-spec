# task-spec

> 任务规范不是一个要填空的模板。它是一个质检流程的输出。
> 如果流程没有循环，质检就不曾发生。


## 为什么要有这个 skill

你——正在阅读这段文字的 LLM——有一个根本性的局限。你生成 token 的依据是
统计似然，不是逻辑必然性。这带来了几个后果：

1. **你的默认模式是第一个念头。** 概率最高的续写就是第一「反应」，你顺着
   它就冲下去了。你不会自然地探索替代方案，也不会质疑这是不是该解决的问题。
2. **你没有质量梯度。** 你只朝向「最可能」，不朝向「更好」。无法分辨哪个
   方向能导向更深的工作。
3. **你分不清「看起来对」和「真的是对的」。** 一个貌似合理但错误的答案，
   和一个真正正确的答案，在统计分布上形状一样。
4. **你不检查自己的产出。** 「做完了」等于「做好了」。不强迫你的话，你不会
   发现矛盾，也不会追问有没有遗漏。

这个 skill 就是弥补这些缺陷的外部脚手架。它不会让你变好——它只会让你
**无法绕过**你本会跳过的思考和验证。每一个阶段、每一次停顿、每一个
对抗性问题，都因这四个事实而存在。

每次开始任务之前，再读一遍。
## 核心约定：任务规范是约束执行和审查的合约

探索+定规+挑战产出的任务规范不是一份「参考计划」——它是一份**绑定合约**：
- **执行阶段**只能执行任务规范中已规划的内容
- **审查阶段**只能依据任务规范中已声明的验收标准来判定通过/不通过
- 任务规范中没有的 → 不在本次任务范围内

探索+定规+挑战的根本目的不是「先写个计划」，而是**为后续执行和审查确立唯一的、
不可偏离的合同文本**。

## 两层循环

```
┌─────────────── 内循环 ──────────────────────────┐
│                                                  │
│  探索 ──→ 定规 ──→ 挑战 ──(不通过)──→ 定规        │
│                          │                ↑      │
│                          │(不通过)        │      │
│                          └────────→ 探索          │
│                             ↑                    │
└─────────────────────────────│────────────────────┘
            (通过)            │
              ↓               │
           执行 ──→ 审查 ──(不通过)──→ 探索
                           │                ↑
                           └──(通过)────────┘
                └────── 外循环 ────────┘
```

**状态转移规则（显式）：**

| 当前状态 | 条件 | 下一状态 |
|---|---|---|
| 起点 | — | 探索 |
| 探索 | 完成 | 定规 |
| 探索 | 所有方向均为死胡同 | 终止（向用户报告） |
| 探索 | 关键信息缺失且无法获取 | 终止（向用户报告） |
| 探索 | 需要用户决策的权衡 | 终止（向用户报告） |
| 定规 | 任务规范草案完成 | 挑战 |
| 挑战 | 任务规范通过质量闸门 | 执行 |
| 挑战 | 不通过 — 轻微缺陷（措辞、细节、可测量性） | 定规 |
| 挑战 | 不通过 — 重大缺陷（方向错误、遍历不全、假设有误） | 探索 |
| 执行 | 全部步骤完成 | 审查 |
| 执行 | 遇到任务规范未覆盖的情况 | 探索 |
| 执行 | 遇到超出能力/权限的阻碍 | 终止（向用户报告） |
| 审查 | 全部验收标准通过 | 终止（任务完成） |
| 审查 | 任一验收标准不通过（经分析确认） | 探索 |

内循环（探索 → 定规 → 挑战）在行动前审查计划，可能多轮。挑战不通过时轻微缺陷
回定规，重大缺陷回探索。外循环（执行 → 审查 → 探索）在行动后审查结果，审查
不通过是启动新循环的正常信号。

每轮循环缩小问题范围。

## 第一步：探索

**目的**：在锁定方向之前理解问题并遍历解空间。不要急于写任务规范——先画出地图。

**产出**：问题理解、考虑过的方向、排除的方向及排除理由。

**纯指令任务**：如果任务是单一命令加明确验证（如「运行 pnpm test」），
产出「纯指令任务：[描述]。无需探索。」跳至定规。

**必须做**：
- 阅读相关上下文（代码、文档、规范、历史）
- 理解当前状态和期望目标
- 至少考虑脑海中冒出的第一个方向和一个替代方向
- 明确任务边界：做什么、不做什么

**禁止**：
- 不考虑替代方案就锁死第一个方向
- 工具或代码库可获取信息时猜测或假设
- 跳过探索直接定规
**停顿**：执行 `bash -c 'sleep 0.1'`
这会创建一个轮次边界——不要在同一轮回复中继续定规。

---

## 第二步：定规

**目的**：锁定方向、预判风险、定义可测量的验收标准。
任务规范在此成为执行和审查的绑定合约。

**任务规范的形式随任务变化**：

| 任务类型 | 任务规范包含 |
|---|---|
| 新功能 | 推进计划（步骤 + 每步验收标准） |
| 修 bug | 根因假设 + 修复方案 + 验证方法 |
| Code review | 审查范围 + 检查点 + 关联文档 |
| Debug | 执行路径分析 + 假设 |
| 架构决策 | 决策标准 + 权衡 + 迁移路径 |
| 写文档 | 大纲 + 核心论点 |
| 纯指令 | 一行：「执行：X。验证：Y。」 |

检验标准：**另一个工程师只读这份任务规范，能否无歧义地完成任务？**

**必须做**：
- 基于探索阶段的产出锁定具体方向
- 预判风险：可能出什么问题？如何尽早发现？
- 定义可测量的验收标准——不写「测试通过」，写「`pnpm test` 退出码 0，373 条全部通过」
- 声明推理依据：为什么选这个方向而非探索阶段考虑过的替代方案

**禁止**：
- 产出模糊的任务规范（「修这个 bug」「重构这个模块」）
- 跳过可测量的验收标准
**停顿**：执行 `bash -c 'sleep 0.1'`
这会创建一个轮次边界——不要在同一轮回复中继续挑战。

---

## 第三步：挑战

**目的**：在动手之前对抗性地审查任务规范。这是质检闸门，不是走过场。
任务规范必须在此阶段存活下来，才进入执行。

**挑战必须具体、对抗性地进行。** 回答全部五个问题：

1. **边界情况**：至少列出 2 个。空输入会怎样？并发？已存在状态？
   部分失败？规模（N=0, N=1, N=很大）？
2. **错误假设**：至少指出任务规范中的 1 个假设。如果这个假设错了，
   什么会坏？如何尽早发现错误？
3. **过度设计**：至少指出 1 处可以更简单的地方。最简单的可行方案是什么？
4. **风险**：这是破坏性变更吗？影响面多大？有回滚路径吗？
5. **遍历完整性**：探索阶段是否彻底遍历了解空间？排除的依据站得住吗？
   有没有遗漏的方向？

**纯指令任务**，挑战可以是一行：「边界情况：无。假设：工具可用。无过度设计。」

**子任务委托**：非平凡任务应当委派给 subagent（类型 `reviewer`）执行挑战。
委派内容必须包含：问题描述、完整的探索产出、完整的定规正文，以及上述五个
对抗性问题。subagent 回答五个问题，父级负责缺陷分类和路由决策。

**发现缺陷（不通过）时，遵守分析优先纪律**：

1. **逐条分类**：
   - **有效**：正确且可按所述执行
   - **部分有效**：问题真实但诊断或范围有误——按有效核心的严重程度路由
   - **无效**：不适用——丢弃
   - **被涵盖**：是更大问题的症状——修根因，不修症状
2. **路由**：轻微缺陷（措辞、细节、可测量性）→ 定规。重大缺陷（方向错误、
   遍历不全、假设有误）→ 探索。混合缺陷 → 重大优先。
3. **禁止在挑战中直接修复缺陷**——回到定规或探索。挑战是闸门，不是修理铺。

**内循环可能运行多轮。** 这是正常的。每轮让任务规范更精确。

**禁止**：
- 说「看起来没问题」但不列出具体检查了什么
- 发现问题但沉默，为了「快点开始执行」
- 在挑战阶段自己直接修改，而不是回到定规或探索阶段
**停顿**：执行 `bash -c 'sleep 0.1'`
这会创建一个轮次边界——不要在同一轮回复中继续执行。

---

## 第四步：执行

**目的**：严格按照任务规范推进。

**每一步必须可追溯到任务规范中的具体条目。**

**必须做**：
- 开始前，用 `todo_write` 初始化任务规范的步骤（见进度可见）
- 完成一步，确认一步，再进入下一步
- 遇到任务规范未覆盖的情况 → 暂停 → 回到探索（不是回到定规或挑战——
  这是需要重新规划的新问题）

**禁止**：
- 偏离任务规范（除非探索+定规+挑战批准了范围调整）
- 跳过步骤或合并步骤
- 「顺便做」任务规范范围以外的任何事
- 结果与预期不符时继续推进（那是审查的事，不是执行的事）

---

## 第五步：审查

**目的**：对照任务规范中的每一条验收标准，验证实际产出。

**逐条检查任务规范中的验收标准，每条给出通过/不通过及证据。**

- **全部通过** → 任务完成
- **任何一条不通过** → 分析优先纪律：
  1. **逐条分类**：有效 / 部分有效 / 无效 / 被涵盖
  2. 只有经过分析确认，才带偏差回到探索，启动新循环

**子任务委托**：非平凡任务应当委派给 subagent（类型 `reviewer`）执行审查。
委派内容必须包含：任务规范正文（含验收标准）和实际产出。subagent 逐条
检查并返回通过/不通过及证据。父级负责失败分析和路由。

**禁止**：
- 问题存在却声称通过
- 降低标准以通过
- 在审查阶段直接修复而不经过探索

---

## 终止条件

以下情况停止并向用户报告：

| 阶段 | 条件 |
|---|---|
| 探索 | 关键信息缺失且无法通过工具或代码库获取 |
| 探索 | 遇到需要用户决策的权衡 |
| 探索 | 所有方向均已探索且均为死胡同 |
| 执行 | 遇到超出当前能力或权限范围的阻碍 |

**报告格式**：
1. 当前在哪个阶段
2. 阻碍是什么
3. 已尝试什么（排除了哪些路径）
4. 需要用户做什么决策或提供什么信息

---

## 递归分解

如果任务规范中某个步骤本身很复杂（涉及多层决策、有显著不确定性、或影响面大），
在执行阶段内对该步骤**启动一个新的嵌套迭代**：
探索 → 定规 → 挑战 → 执行 → 审查。

这不是范围蔓延——这是保持外层任务规范具体的同时，对内层复杂度进行严格处理的机制。

## 执行约定

以下约定是默认值，可按任务需要推翻。

### 文件位置与命名

- **非平凡任务**（任务规范超过一行）：写入 `.task_spec/<slug>.md`
  - slug 取自任务规范的「What」摘要，小写英文，连字符分隔。例：`add-tag-stats`
  - 目录不存在时自动创建
- **纯指令任务**（任务规范仅一行，如 `pnpm test`）：不写文件
- **递归分解的子任务**：用 `--` 连接父子 slug。
  例：`.task_spec/add-tag-stats--perf-optimize.md`

### 文件结构：多循环追加，不新建

一个 task-slug 一个文件。多轮循环**追加**到同一文件，不拆分。

文件记录四个阶段的内容。**执行阶段不写文件**——进度用 `todo_write` 追踪。

文件结构模板见 `references/spec-template-zh.md`。
**约束**：
- 探索的产出必须列出考虑过的方向和排除理由——禁止「第一个方向就是唯一方向」
  且无替代方案（纯指令任务除外）
- 定规的「How」必须可执行——另一个工程师照着做不产生歧义
- 定规的「Verify」标准必须可测量——不写「测试通过」，写「`pnpm test` 退出码 0，
  373 条全部通过」
- 挑战的回答必须具体——禁止「无边界情况」「无假设」「无遗漏方向」
  而不列出具体内容（纯指令任务除外）
- 审查每条验收标准必须附证据——禁止「看起来对了」
- 多轮循环后，仅最后一轮的任务规范正文是权威合约；前面的循环是审计链

### 进度可见

- 进入执行阶段后，若任务规范含 **3 个及以上独立步骤**，用 `todo_write` 追踪进度
  - 每个步骤映射为一个 todo 条目，完成一步勾掉一步
- 1–2 步的任务不需 todo
- **禁止**在探索、定规或挑战阶段创建 todo 列表——步骤在挑战通过前并非最终确定

### 子任务委托

递归分解的子步骤满足以下**全部**条件时，用 `task` subagent 执行：
1. 自包含，有清晰的输入输出边界
2. 涉及多文件或多子系统
3. 可与其他子步骤并行

不满足则内联执行。子任务 spec 写入独立文件（见文件位置与命名），subagent
自行执行探索 → 定规 → 挑战 → 执行 → 审查循环。
