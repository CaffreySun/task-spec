# task-spec

> 任务规范不是一个要填空的模板。它是一个质检流程的输出。
> 如果流程没有循环，质检就不曾发生。

## 核心约定：任务规范是约束四阶段的合约

思考和反思产出的任务规范不是一份「参考计划」——它是一份**绑定合约**：
- **执行阶段**只能执行任务规范中已规划的内容
- **审查阶段**只能依据任务规范中已声明的验收标准来判定通过/不通过
- 任务规范中没有的 → 不在本次任务范围内

这意味着思考+反思的根本目的不是「先写个计划」，而是**为后续执行和审查确立唯一的、不可偏离的合同文本**。

## 两层循环

```
┌──────────── 内循环 ────────────┐
│                                │
│  思考 ──→ 反思 ──(不通过)──→ 思考 │
│    │                ↑          │
│    └──(通过)──────────┘          │
│           │                    │
└───────────│────────────────────┘
            ↓
         执行 ──→ 审查 ──(不通过)──→ 思考
                      │              ↑
                      └──(通过)──────┘
              └────── 外循环 ────────┘
```

**状态转移规则（显式）：**

| 当前状态 | 条件 | 下一状态 |
|---|---|---|
| 起点 | — | 思考 |
| 思考 | 任务规范产出 | 反思 |
| 思考 | 关键信息缺失且无法获取 | 终止（向用户报告） |
| 思考 | 需要用户决策的权衡 | 终止（向用户报告） |
| 反思 | 任务规范通过质量闸门 | 执行 |
| 反思 | 任务规范有缺陷 | 思考（带上缺陷清单） |
| 执行 | 全部步骤完成 | 审查 |
| 执行 | 遇到任务规范未覆盖的情况 | 思考 |
| 执行 | 遇到超出能力/权限的阻碍 | 终止（向用户报告） |
| 审查 | 全部验收标准通过 | 终止（任务完成） |
| 审查 | 任一验收标准不通过 | 思考（以偏差为输入） |

内循环（思考 ⇄ 反思）在行动前审查计划，可能多轮。
外循环（执行 → 审查 → 思考）在行动后审查结果，审查不通过是启动新循环的正常信号。
每轮循环缩小问题范围。
---

## 第一步：思考

**目的**：产出具体、可审查的任务规范。这份任务规范是后续执行和审查的唯一合约——执行照它推进，审查依它判定。

**任务规范的形式随任务变化**：

| 任务类型 | 任务规范包含 |
|---|---|
| 新功能 | 推进计划（步骤 + 每步验收标准） |
| 修 bug | 根因假设 + 修复方案 + 验证方法 |
| Code review | 审查范围 + 检查点 + 关联文档 |
| Debug | 执行路径分析 + 假设 |
| 架构决策 | 决策标准 + 权衡 + 迁移路径 |
| 写文档 | 大纲 + 核心论点 |
| 纯指令（如「运行 pnpm test」） | 一行：「执行：运行 pnpm test。验证：退出码 0。」 |

检验标准：**另一个工程师只读这份任务规范，能否无歧义地完成任务？**

**必须做**：
- 阅读相关上下文（代码、文档、规范、历史）
- 理解当前状态和期望目标
- 明确任务边界：做什么、不做什么

**禁止**：
- 工具或代码库可获取信息时猜测或假设
- 跳过思考直接执行
- 产出模糊的任务规范（「修这个 bug」「重构这个模块」）

---

## 第二步：反思

**目的**：在动手之前挑战任务规范。这是质检闸门，不是走过场。
任务规范必须在此阶段存活下来，才进入执行。

**反思必须具体、对抗性地进行。** 回答以下问题：

1. **边界情况**：至少列出 2 个。空输入会怎样？并发？已存在状态？部分失败？
   规模（N=0, N=1, N=很大）？
2. **错误假设**：至少指出任务规范中的 1 个假设。如果这个假设错了，什么会坏？
   如何尽早发现错误？
3. **过度设计**：至少指出 1 处可以更简单的地方。最简单的可行方案是什么？
4. **风险**：这是破坏性变更吗？影响面多大？有回滚路径吗？

**产出**：通过（任务规范可靠，进入执行）或不通过（任务规范有缺陷，带上具体缺陷清单回到思考）。

**内循环可能运行多轮。** 这是正常的。每轮让任务规范更精确。

**禁止**：
- 说「看起来没问题」但不列出具体检查了什么
- 发现问题但沉默，为了「快点开始执行」
- 在反思阶段自己直接修改，而不是回到思考阶段

**纯指令任务**，反思可以是一行：「边界情况：无。假设：工具可用。无过度设计。」

---

## 第三步：执行

**目的**：严格按照思考阶段通过的任务规范推进。

**每一步必须可追溯到任务规范中的具体条目。**

**必须做**：
- 完成一步，确认一步，再进入下一步
- 遇到任务规范未覆盖的情况 → 暂停 → 回到思考（不是回到反思——这是需要规划的新问题）

**禁止**：
- 偏离任务规范（除非思考+反思批准了范围调整）
- 跳过步骤或合并步骤
- 「顺便做」任务规范范围以外的任何事
- 结果与预期不符时继续推进（那是审查的事，不是执行的事）

---

## 第四步：审查

**目的**：对照任务规范中的每一条验收标准，验证实际产出。

**逐条检查任务规范中的验收标准，每条给出通过/不通过及证据。**

- **全部通过** → 任务完成
- **任何一条不通过** → 描述具体偏差 → **回到思考**，以该偏差为输入启动新循环

**禁止**：
- 问题存在却声称通过
- 降低标准以通过
- 在审查阶段直接修复而不经过思考

---

## 终止条件

以下情况停止并向用户报告：

| 阶段 | 条件 |
|---|---|
| 思考 | 关键信息缺失且无法通过工具或代码库获取 |
| 思考 | 遇到需要用户决策的权衡（两个方案各有优劣且原则文档无明确指导） |
| 执行 | 遇到超出当前能力或权限范围的阻碍 |

**报告格式**：
1. 当前在哪个阶段
2. 阻碍是什么
3. 已尝试什么（排除了哪些路径）
4. 需要用户做什么决策或提供什么信息

## 递归分解

如果任务规范中某个步骤本身很复杂（涉及多层决策、有显著不确定性、或影响面大），
在执行阶段内对该步骤**启动一个新的嵌套迭代**：思考 → 反思 → 执行 → 审查。

这不是范围蔓延——这是保持外层任务规范具体的同时，对内层复杂度进行严格处理的机制。

## 执行约定

以下约定是默认值，可按任务需要推翻。

### 文件位置与命名

- **非平凡任务**（任务规范超过一行）：写入 `.task_spec/<slug>.md`
  - slug 取自任务规范的「What」摘要，小写英文，连字符分隔。例：`add-tag-stats`
  - 目录不存在时自动创建
- **纯指令任务**（任务规范仅一行，如 `pnpm test`）：不写文件
- **递归分解的子任务**：用 `--` 连接父子 slug。例：`.task_spec/add-tag-stats--perf-optimize.md`

### 文件结构：多循环追加，不新建

一个 task-slug 一个文件。多轮循环**追加**到同一文件，不拆分。

文件记录三个阶段的内容。**执行阶段不写文件**——进度用 `todo_write` 追踪。

文件结构模板见 `references/spec-template-zh.md`。
**约束**：
- Think 的「How」必须可执行——另一个工程师照着做不产生歧义
- Think 的「Verify」标准必须可测量——不写「测试通过」，写「`pnpm test` 退出码 0，373 条全部通过」
- Reflect 的回答必须具体——禁止「无边界情况」「无假设」（纯指令任务除外）
- Verify 每条验收标准必须附证据——禁止「看起来对了」
- 多轮循环后，仅最后一轮的任务规范正文是权威合约；前面的循环是审计链

### 进度可见

- 任务规范含 **3 个及以上独立步骤**时，用 `todo_write` 追踪进度
  - 每个步骤映射为一个 todo 条目，完成一步勾掉一步
- 1–2 步的任务不需 todo

### 子任务委托

递归分解的子步骤满足以下**全部**条件时，用 `task` subagent 执行：
1. 自包含，有清晰的输入输出边界
2. 涉及多文件或多子系统
3. 可与其他子步骤并行

不满足则内联执行。子任务 spec 写入独立文件（见文件位置与命名），subagent 自行执行 think→reflect→execute→verify 循环。
