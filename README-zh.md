# task-spec

防止 AI 在思考之前就动手执行——通过在每一次任务上构建对抗性质控循环。

如果把一个基于 LLM 的 AI Agent 当作一个人来看，这个人有几个特点：
知识极其渊博，智商很高，但做事敷衍。它产出的答案看起来都对——
合理、有条理、语气自信——但它不会停下来确认这些答案到底对不对。
足够聪明把活干了，但默认选择最省力的干法。

这不是态度问题。背后的机制比"懒"深刻得多。

## 问题

LLM 没有内建的验证循环。它基于统计似然生成下一个 token，而非逻辑必然性。
一个数学上正确的证明和一个看起来对但实际错的论证，对模型来说没有区别——
两者都存在于训练分布中，模型无法从内部区分。

这是所有"AI 做事敷衍"背后的机制：

- **不会停下来质疑方向**——没有基于正确性评估来自我打断的机制
- **默认"直接开始干活"**——从 prompt 到输出的最短路径就是最可能的路径
- **编造看似正确但实则错误的内容**——看似正确 = 统计上高频；验证正确性是模型不具备的能力
- **长会话中前后矛盾或跑偏**——注意力机制衰减；没有验证步骤去捕捉矛盾

训练数据质量解决不了这个问题。模型只能复现正确工作产出的文本形态，学不到
产生这些文本之前的推理过程。人类在落笔之前做的验证对模型是不可见的——模型能
看到的只有输出，没有过程。

## 能修吗？

这是架构性限制，不是训练数据的问题。next-token prediction 这个核心机制，
在设计上就把"正确性"排除在生成逻辑之外。再好的数据、再聪明的 prompt、
再多的 RLHF 对齐，也加不进去一个内建的验证循环。

**模型层**：未来有可能。process reward model、能在生成中途暂停-评估-修正的
推理架构，这些方向在向真正的自验证靠近。但这是研究前沿——今天还不能作为
工程依赖。

**系统层**：现在就能缓解。验证不一定非得长在模型里面。外部脚手架——
强制的对抗性挑战、绑定合约、证据审查——可以弥补模型缺失的能力。
task-spec 走的就是这条路。

task-spec 不应该追求"让模型变好"。它该做的事是让模型**逃不掉审查**。
## 答案

task-spec 构造一个人造的对抗环境，弥补这一缺失的能力。它不是让模型学会验证——
是让模型**无法绕过**验证。

每次动手之前，AI 产出可审查的 **spec**（包含推理依据、执行步骤、可测量的验收
标准）。spec 是一份绑定合约——执行照它推进，审查依它判定。没有 spec，不能执行。

spec 产出后立即经受对抗性挑战（边界情况、错误假设、过度设计、风险）。
挑战不通过就回到思考，循环直到 spec 存活。

执行完成后，逐条检查验收标准，每条必须附证据。任何一条不通过就回到思考，
以偏差为新输入开启下一轮循环。

spec 可以只有两行。不能是零行。


## 工作原理

**内循环**（思考 ⇄ 反思）——AI 产出带推理链条的 spec，然后挑战它。
空输入会怎样？哪个假设可能是错的？有没有更简单的方案？影响面多大？
spec 有缺陷就回思考修复。

**外循环**（执行 → 审查 → 思考）——AI 严格照 spec 推进，完成后逐条
验收标准检查，每条附证据。不通过就回思考。

每次的 spec 写入 `.task_spec/<slug>.md`，留下审计链。
