# task-spec

防止 AI 在思考之前就动手执行——通过在每一次任务上构建对抗性质控循环。

如果把一个基于 LLM 的 AI Agent 当作一个人来看，这个人有几个特点：
知识极其渊博，智商很高，但做事敷衍。它产出的答案看起来都对——
合理、有条理、语气自信——但它不会停下来确认这些答案到底对不对。
足够聪明把活干了，但默认选择最省力的干法。

这不是态度问题。背后的机制比"懒"深刻得多。

## 问题

两个同等聪明的人做同一件事，一个做得好一个做得不好，差别在哪？
好的人在动手前想透了——探索替代方案、质疑前提、预判问题。
差的人脑子里冒出第一个念头就开干，目标是尽快收工。

LLM 的默认模式正是那个"差的人"。next-token prediction 没有"停下来评估"
这个概念。概率最高的续写就是第一反应，模型顺着它就冲下去了。

这体现在两个维度：

**动手前——思之不足。** 模型抓住冒出来的第一个方向就走。不探索替代方案，
不追问"这真的是该解决的问题吗"，不预判后续影响。把"有个答案"当成
"答案是对的"。

**动手后——行之不验。** 模型不检查自己的产出。"做完了"等于"做好了"。
不会发现前后矛盾，不会拿输出和初衷比对，不会问有没有遗漏。

训练数据质量修不了这两个维度。更好的数据只能让模型知道正确的产出
*长什么样*，学不到产出之前的思考过程。人类的停顿、怀疑、被放弃的
第一版——这些验证过程在最终输出里是看不见的。

## 能修吗？

这是架构性限制，不是训练数据的问题。next-token prediction 这个核心机制，
在设计上就把"正确性"排除在生成逻辑之外。再好的数据、再聪明的 prompt、
再多的 RLHF 对齐，也加不进去一个内建的验证循环。

**模型层**：未来有可能。process reward model、能在生成中途暂停-评估-修正的
推理架构，这些方向在向真正的自验证靠近。但这是研究前沿——今天还不能作为
工程依赖。

**系统层**：两个维度不是在难度上不同，是在性质上不同。

验证有**外部值函数**。"输出和 spec 对得上吗"是客观问题。可以用绑定合约和
证据审查机械地强制执行。标准在模型外面。

思考没有这个东西。"这个方案对吗""这够好了吗"需要**内部值函数**——一个指向
"更好"的方向感。模型没有这个梯度。next-token prediction 只朝向"最可能"，
不朝向"更好"。模型不是懒才停在浅层——是无法分辨哪个方向更深。
人类类比是态度：一个想把事做好的人，自然会探索、比较、质疑、推翻重来。
模型没有对等的驱动力，它不会"想"任何事。

这就是为什么两个维度不能用同一套机制。验证可以用外部合约强制执行。
思考只能靠结构去模拟质量梯度——逼模型走一遍高态度的人自然会走的步骤。
## 答案

task-spec 用两套不同的机制分别应对两个维度。

**验证维度——Spec 驱动的执行。**
spec 是绑定合约：执行照它推进，审查依它判定。每条验收标准对照实际产出，
附证据判断通过或不通过。没有 spec，不能执行。验证从外部强制执行——输出
和 spec 对不上就是对不上。

**思考维度——先探索再定 spec。**
锁定方案之前，模型遍历解空间：理解问题本身、考虑可能的方向、排除死胡同。
然后才锁定方向、预判风险、定义可测量的成功标准。spec 是探索的产出，不是
起点。这套结构模拟的是高态度的人自然会做的事——在解空间里走一圈再落点。

探索本身也要经受对抗性挑战。spec 定稿之前，模型直面自身的推理：解空间
遍历完整吗？排除的依据站得住吗？只有活过这轮挑战，spec 才成为执行的合约。

## 工作原理

**内循环**（思考 ⇄ 反思）——AI 产出带推理链条的 spec，然后挑战它。
空输入会怎样？哪个假设可能是错的？有没有更简单的方案？影响面多大？
spec 有缺陷就回思考修复。

**外循环**（执行 → 审查 → 思考）——AI 严格照 spec 推进，完成后逐条
验收标准检查，每条附证据。不通过就回思考。

每次的 spec 写入 `.task_spec/<slug>.md`，留下审计链。
