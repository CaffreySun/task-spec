# task-spec

防止 AI 在思考之前就动手执行——通过在每一次任务上构建对抗性质控循环。

## 问题

LLM 有两个根深蒂固的问题。一个出在动手之前，一个出在动手之后。

如果把 AI 看成一个聪明但做事敷衍的人——知识渊博，智商很高，产出
看起来都对，但从来不停下来确认到底对不对。给它一个问题，它的第一
反应就是冲上去开干。

这不是态度问题。背后的机制比"懒"深刻得多。

LLM 生成文本的方式是，一次预测一个词。每一步回答的都是"下一个词
统计上最像什么"，而不是"下一个词怎么才对"。这意味着模型的第一
直觉永远是最可能的路径，不是最谨慎的路径。一个正确的答案和一个
貌似合理但错误的答案，在统计分布上形状一样。模型无法从内部区分。

这体现在两个维度：

**动手前——思之不足。** 模型抓住冒出来的第一个方向就走。不探索
替代方案，不追问"这真的是该解决的问题吗"，不预判后续影响。
把"有个答案"当成"答案是对的"。

**动手后——行之不验。** 模型不检查自己的产出。"做完了"等于
"做好了"。不会发现前后矛盾，不会拿输出和初衷比对，不会问有没有
遗漏。

更好的训练数据修不了这两个维度。更好的数据只能让模型知道正确的
产出*长什么样*——学不到产出之前的思考过程。人类的犹豫、怀疑、
被放弃的第一版——写下来之前在脑子里跑过的验证——对模型来说是
不可见的。模型看到的只有最终结果，从来没有过程。

## 能修吗？

这是架构性限制，不是训练数据的问题。生成机制——next-token
prediction——在设计上就把"正确性"排除在考察范围之外。再好的
数据、再聪明的 prompt、再多的 RLHF 对齐，也加不进去一个内建的
验证循环。

**模型层**：未来有可能。process reward model、能在生成中途暂停-
评估-修正的推理架构，这些方向在向真正的自验证靠近。但这是研究
前沿——今天还不能作为工程依赖。

**系统层**：两个维度不是在难度上不同，是在性质上不同。

验证有**外部值函数**。"输出和 spec 对得上吗"是客观问题。可以用
绑定合约和证据审查机械地强制执行。标准在模型外面。

思考没有这个东西。"这个方案对吗""这够好了吗"需要**内部值函数**
——一个指向"更好"的方向感。模型没有这个梯度。next-token
prediction 只朝向"最可能"，不朝向"更好"。模型不是懒才停在浅层
——是无法分辨哪个方向更深。人类的对应概念是态度：一个想把事
做好的人，自然会探索、比较、质疑、推翻重来。模型没有对等的驱动
力。它不会"想"任何事。

这就是为什么两个维度需要不同的策略。验证可以用外部合约强制执行。
思考只能靠结构去模拟质量梯度——逼模型走一遍高态度的人自然而然
会走的步骤。

## 答案

task-spec 用一个五阶段流程分别应对这两个维度：

**探索 → 定规 → 挑战 → 执行 → 审查**

**探索**——遍历解空间。理解问题、考虑方向、排除死胡同。先别急着
锁定。

**定规**——锁定方向。定义可测量的验收标准、预判风险、写下推理
依据。spec 是绑定合约：执行照它推进，审查依它判定。

**挑战**——对抗性审查。五个问题：边界情况、错误假设、过度设计、
风险、解空间是否遍历完整。非平凡任务委派给 subagent 执行。没过
这关的 spec 不能进入执行。

**执行**——严格照 spec 推进。每一步都可追溯到 spec。遇到未覆盖
的情况，暂停，回到探索。

**审查**——逐条验收标准对照实际产出，附证据。通过？完成。不通过？
先分析失败是否真实，再回到探索启动新循环。

五个阶段形成两层循环。**内循环**（探索 → 定规 → 挑战）在行动前
打磨方案——轻微缺陷回定规，重大缺陷回探索。**外循环**（执行 →
审查 → 探索）在行动后验证结果——审查不通过触发全新一轮。

每次的 spec 写入 `.task_spec/<slug>.md`，留下审计链。
